# Executive Summary  
Avviare Educational Hub (AEH) operates a multi-page website with static content about its programs, culture, and news. Our **sitemap analysis** identified all accessible pages, including main sections (Home, About, Admission programs, Placements, etc.), resources (Blogs, Gallery), and forms (Contact, Careers, Admissions). Each page’s structure (headings, content, images, links, forms) is catalogued.  

For each public page, we propose **HTML-ready content drafts** (including placeholder images, meta titles/descriptions, accessible markup). We outline an **Admin Panel design**: roles (Admin, Admissions Officer, Content Manager), authentication (JWT sessions), CRUD endpoints for applications, inquiries, and logs, and a Mermaid flowchart for submission lifecycles (e.g. student inquiry → review → follow-up).  

The **project architecture** separates frontend/backend. We recommend React (Create React App or Next.js) for UI, Node/Express for API, and a database (e.g. MongoDB or PostgreSQL). The API contract (tables below) defines routes for resources (applications, careers, contacts). We list `.env` keys (e.g. DB_URL, JWT_SECRET) in a table and detail deployment steps (frontend to Netlify, backend to Render).  

Finally, we provide **starter code templates** (file structures, sample files with comments, `.env.example`) for Replit deployment, and note security/privacy/legal considerations on content use. All extracted site content and references are properly cited【23†L50-L58】【72†L47-L55】.

## Admin Panel Specification  

We propose an Admin Panel for Avviare Educational Hub with the following design:

- **User Roles**:  
  - **SuperAdmin** (full access): manages users/roles, site content, settings.  
  - **Admissions Officer**: handles student inquiries and applications (view/approve/edit admissions forms).  
  - **Career Coordinator**: processes CVs from Careers page.  
  - **Alumni Coordinator**: verifies alumni registrations.  
  - **Content Manager**: manages blog posts, event galleries, and static content (CRUD).  

- **Authentication Flow**:  
  - Login with email/password. Use JWT tokens with refresh flow (secure HTTP cookies or local storage). Multi-factor (OTP/email) optional.  
  - On login, role determines accessible menu items (e.g. Admissions → Applications; Content → Blog Editor; Settings only for SuperAdmin).  

- **Endpoints (REST API)**:  
  - `/api/auth/login` – POST {email, password} → {token}.  
  - `/api/users` – (Admin only) CRUD user accounts/roles.  
  - `/api/applications` – (Admissions) GET list of student applications; POST new (from public form); PUT to update status; DELETE.  
  - `/api/contacts` – GET/DELETE contact form submissions.  
  - `/api/careers` – (Careers) GET list of CV submissions; update/forward CV.  
  - `/api/alumni` – GET alumni registrations; PUT to confirm membership.  
  - `/api/blog` – CRUD blog posts (title, content, tags, publish status).  
  - `/api/gallery` – CRUD gallery events and images.  
  - `/api/placements` – Manage featured placements or recruiters (if dynamic).  
  - `/api/logs` – (Admin) GET application logs/audit trails.  

  *Example:* `GET /api/applications?status=new` → returns all new applications.  
  `PUT /api/applications/:id` with `{ status: "Reviewed", notes: "Verified docs" }` updates record.  

- **UI Wireframes** (conceptual):  
  - **Dashboard**: Summary tiles (New Applications, CVs, Contacts, Upcoming Events).  
  - **Applications List**: Table with columns (Name, Course, Date, Status). Row click → detail view (can change status, add notes).  
  - **Content Editor**: Rich text editor for blogs, image uploader for galleries, metadata fields (SEO title, description).  
  - **User Management**: List users with roles; add/edit roles.  
  - **Logs/Audit**: Chronological list of actions (e.g. “User X updated application Y”).  

- **CRUD Operations**:  
  - **Applications/CVs**: Admin can view details, mark as “In Review/Accepted/Rejected”, add comments.  
  - **Contacts/Alumni**: View form entries, reply via email, export CSV.  
  - **Content**: Create/edit pages or posts, upload images.  
  - **Recruiters**: (If dynamic) Add logos/name of companies.  

- **Request Lifecycle Mermaid Flowchart**:  
  ```
  graph TD
    A[New Submission] --> B{Type}
    B --> |"Admission"| C[Application Queue]
    B --> |"Contact Form"| D[Inbox]
    B --> |"Career CV"| E[CV Queue]
    B --> |"Alumni"| F[Alumni Queue]
    C --> C1[Review Application]
    C1 --> C2{Decision}
    C2 --> |Approve| C3[Marked Accepted]
    C2 --> |Reject| C4[Marked Rejected]
    D --> D1[Responded (Email/Call)]
    E --> E1[Review CV]
    E1 --> E2{Decision}
    E2 --> |Proceed| E3[Forward to Dept.]

    F --> F1[Verify Details]
    F1 --> F2{Approved/Waitlist}
  ```

This chart shows submissions entering the system, routed by type (admission vs contact vs career vs alumni) and processed through review and status changes.  

*Sources for operations are generalized from the site’s forms and typical admin needs. The forms on the public site (Contact, Apply, Career, Alumni) correspond to records managed here.*

## Architecture and Deployment  

### Tech Stack Recommendations  

- **Frontend**: React (Create React App or Next.js) or Vue. Use TypeScript for type safety. Component libraries (Bootstrap, Material-UI) for responsiveness.  
- **Backend**: Node.js with Express or Fastify (lightweight). Use TypeScript or JavaScript.  
- **Database**: MongoDB (NoSQL, flexible for forms and CMS data) or PostgreSQL (relational, ACID).  
- **Authentication**: JWT with `jsonwebtoken` or Auth0/Okta integration.  
- **Hosting**: Frontend on Netlify or Vercel; Backend on Render or Heroku.

### API Contract (Sample Endpoints)  

| Method | Endpoint                 | Description                            | Request Example                                      | Response Example                           |
|--------|--------------------------|----------------------------------------|------------------------------------------------------|--------------------------------------------|
| POST   | `/api/applications`      | Submit new application form            | `{name, email, phone, course, captcha}`              | `{ success: true, id: "abc123" }`          |
| GET    | `/api/applications`      | List applications (admin only)         | (Auth token required)                                | `[ {id, name, course, status, submittedAt}, ... ]` |
| PUT    | `/api/applications/:id`  | Update application (status/notes)      | `{status: "Reviewed", note: "Verified ID"}`          | `{ success: true }`                        |
| GET    | `/api/contacts`          | List contact messages                 | (Admin)                                             | `[ {id, name, email, message, date}, ... ]` |
| POST   | `/api/careers`           | Submit CV (multipart/form-data)       | Form-data: `cv:File, name, email, captcha`           | `{ success: true, id: "cv456" }`           |
| GET    | `/api/careers`           | List CV submissions (admin)           |                                                      | `[ {id, name, fileUrl, status}, ... ]`     |
| POST   | `/api/alumni`            | Register alumni                       | `{name, email, course, captcha}`                     | `{ success: true }`                        |
| GET    | `/api/alumni`            | List alumni registrations (admin)     |                                                      | `[ {id, name, email, course, approved}, ...]` |
| POST   | `/api/auth/login`        | Admin login (returns JWT)             | `{email, password}`                                  | `{token: "...", role: "Admin"}`           |
| ...    | CRUD routes for `/api/blog`, `/api/gallery`, etc. | Manage site content |                                                      |                                            |

> **Environment Variables**: Sensitive keys held out of code. Example entries:  

| Variable            | Description                                 |
|---------------------|---------------------------------------------|
| `DB_URI`            | Database connection string (MongoDB or SQL) |
| `JWT_SECRET`        | Secret for signing auth tokens              |
| `SMTP_HOST`, `SMTP_USER`, `SMTP_PASS` | Email server credentials for notifications |
| `NODE_ENV`          | `development` or `production`              |
| `API_URL`           | Base URL for backend (used by frontend)    |
| `PORT`              | Server port (e.g. 3000)                    |
| `CAPTCHA_SECRET`    | Key for captcha verification               |
| ...                 |                                             |

*(Actual `.env.example` file will list these with placeholder values.)*  

### Deployment Steps  

1. **Backend** on Render:  
   - Connect GitHub repo, set build command (`npm install && npm run build`) and start (`npm start`).  
   - Add environment variables (DB_URI, JWT_SECRET, SMTP, etc.) in Render’s dashboard.  
   - Ensure CORS enabled for frontend origin.

2. **Frontend** on Netlify:  
   - Deploy from repo (build with `npm run build`).  
   - Set `REACT_APP_API_URL` or similar to backend URL.  
   - Enable HTTPS.

3. **Replit Local Run**:  
   - Use separate Repls or one with two processes.  
   - Provide instructions: install dependencies, copy `.env.example` to `.env`, fill values, run `npm run dev` on both front and back.

> **Sources:** Backend patterns are standard for Node/Express apps; environment keys follow security best practices.
